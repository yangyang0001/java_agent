# 四个 Agent 框架

对应 Hello-Agents 教程第六章「框架开发实践」。本章原教程用 Python 演示了 4 个主流多智能体框架，
本项目在 `chapter_06` 下用 Java 分别还原了它们各自最核心的编排思想(对话轮转 / 消息广播 / 角色扮演 / 状态图)，
不依赖这些框架本身(Java 生态里也没有对等的成熟实现)。

## 1. AutoGen(`autogen` 包)

**是什么**：微软出品的多智能体对话框架，核心思想是"用对话驱动协作"——把多智能体协作建模成一场群聊，
每个智能体是群聊里的一个"发言人"，通过 `RoundRobinGroupChat` 等策略决定发言顺序。

**优点**
- 心智模型简单：协作 = 对话，容易理解和调试(直接看聊天记录就知道哪一步出了问题)。
- 官方生态成熟，支持人类介入(Human-in-the-loop)、工具调用、多种群聊策略(轮转、按需发言等)。

**缺点**
- 群聊的发言顺序/终止条件本质是"涌现"出来的，复杂任务下协作路径不够可控，容易发言跑题或提前收敛。
- 多轮对话上下文会快速膨胀，token 成本高。

**怎么用**：定义几个带角色 System Prompt 的 Agent，塞进一个 GroupChat 容器，指定终止条件(如某角色说出
"APPROVE"/"TERMINATE")，然后调用 `run`，框架自动按策略调度发言顺序。

**常见场景**：需要多个"专业角色"头脑风暴或走一遍评审流程的任务，例如本项目案例——
产品经理 → 工程师 → 审查员 轮流发言，直到审查员通过为止(见 `AutoGenAgent`)。

## 2. AgentScope(`agentscope` 包)

**是什么**：阿里出品的工程化多智能体平台，核心思想是"消息驱动架构"——智能体之间不直接互相调用方法，
而是都连接到一个 `MsgHub`(消息枢纽)，发言即广播，其他成员从枢纽里读消息。

**优点**
- 消息通信天然解耦，容易扩展到高并发、分布式部署(每个 Agent 可以是独立进程/服务)。
- 支持结构化输出、容错重试等工程化特性，更适合生产级系统而不只是原型验证。

**缺点**
- 心智模型比 AutoGen 更"工程化"，上手成本更高，需要理解消息枢纽、订阅关系等概念。
- 对于简单的两三个角色的协作，用消息总线反而比直接对话调用更繁琐。

**怎么用**：创建一个 `MsgHub`，多个 Agent 加入同一个 Hub；每个 Agent 拿到 Hub 里的历史消息作为上下文
生成自己的发言，再广播回 Hub；由外部循环控制"回合"(如昼/夜阶段)推进。

**常见场景**：需要多个身份/阵营互相博弈、信息不对称的场景，例如本项目案例——
"三国狼人杀"游戏，狼人夜晚私聊选择目标、白天所有存活玩家通过 MsgHub 发言与投票(见 `AgentScopeAgent`)。

## 3. CAMEL(`camel` 包)

**是什么**：一个轻量级的角色扮演框架，核心思想是"Inception Prompting"——用系统提示词把两个智能体分别
"锁定"为指导者(AI User)和执行者(AI Assistant)，指导者只管下指令，执行者只管照做，直到任务完成。

**优点**
- 框架本身极简，几乎不需要额外的调度逻辑，两个角色互相喂话就能自驱动完成任务。
- 角色分工清晰，天然适合"一个人出主意、一个人干活"的双人协作场景。

**缺点**
- 只适合双智能体场景，扩展到 3 个以上角色需要自己搭配额外的协调逻辑。
- 缺少显式的流程控制(比如中途分支、并行)，协作路径完全由对话内容驱动，可控性较弱。

**怎么用**：分别写好指导者和执行者的角色设定 Prompt(包含任务描述和"你只能做 X 不能做 Y"的约束)，
循环调用"指导者生成指令 → 执行者生成方案"，直到指导者输出约定的完成标记(如 `<CAMEL_TASK_DONE>`)。

**常见场景**：需要"专家指导 + 专业执行"这种双角色深度协作的创作类任务，例如本项目案例——
心理学家(指导者)与作家(执行者)合作撰写一本关于"克服拖延症"的电子书(见 `CamelAgent`)。

## 4. LangGraph(`langgraph` 包)

**是什么**：LangChain 团队推出的图编排框架，核心思想是把智能体流程建模成一个有状态的图——
节点(Node)是处理步骤，边(Edge)决定下一步去哪，天然支持循环(不像普通链式调用只能单向前进)。

**优点**
- 显式建模流程(节点+边)，控制力最强，容易加条件分支、重试、回环等复杂逻辑。
- 天然支持"自我反思"类模式：某一步判断结果不满意时可以直接把流程绕回之前的节点重新走。

**缺点**
- 需要先把业务流程抽象成图结构，简单任务用图建模反而增加了前期设计成本。
- 状态(State)在多个节点间传递，节点一多，状态字段容易膨胀、难以维护。

**怎么用**：定义共享状态对象，把每一步业务逻辑写成读写状态的节点函数，用 `addNode`/`addEdge` 搭建
节点关系，用 `addConditionalEdge` 描述"满足什么条件走哪条边"，最后从入口节点开始 `invoke`。

**常见场景**：需要循环/回退的多步骤流程，例如本项目案例——
"理解 → 搜索 → 回答"三步问答助手，当"回答"节点判断信息不足时，会通过条件边绕回"理解"节点重新来一轮
(见 `LangGraphAgent` / `StateGraph`)。

## 对比小结

| 框架 | 协作模式 | 控制方式 | 核心优势 | 典型场景 |
|---|---|---|---|---|
| AutoGen | 对话驱动 | 群聊策略(轮转等) | 简单直观、易调试 | 多角色评审/头脑风暴 |
| AgentScope | 消息驱动 | 消息枢纽订阅广播 | 工程化、易扩展/分布式 | 多阵营博弈、生产级系统 |
| CAMEL | 角色扮演 | 指导者/执行者双角色 | 框架极简、自驱动 | 双人深度协作创作 |
| LangGraph | 状态图 | 节点+边(含条件边) | 流程可控、支持循环 | 有回退/重试的多步骤流程 |

一句话取舍：**越靠左，协作越"自然涌现"、上手越快；越靠右，流程越"显式可控"、适合复杂工程场景。**
